================================================================================
PART 1: FRONTEND LAYER DATABASE & ORM ANALYSIS
================================================================================
The frontend is a React Single Page Application (SPA) built using Vite and TailwindCSS.

1. ORM / Local Database:
   - There is NO client-side database (such as SQLite, IndexedDB wrappers like
     Dexie.js, or PouchDB) deployed in the browser.
   - Consequently, there is NO frontend ORM library.

2. Data Persistence & Communication:
   - Data is stored in memory using React states, context providers (e.g., ThemeContext,
     LanguageContext), or managed temporarily via React Router.
   - Persisted UI options (like user theme preferences: Light/Dark mode) are stored
     directly as simple key-value pairs in the browser's standard `localStorage`
     utility (e.g., `localStorage.getItem("theme")`).
   - The application interacts with the backend database exclusively by making asynchronous
     HTTP requests (GET, POST, PUT, DELETE) using the `Axios` HTTP client library,
     communicating with REST endpoints defined on the Spring Boot backend.

================================================================================
PART 2: BACKEND LAYER ORM & PERSISTENCE ARCHITECTURE
================================================================================
The backend is a Spring Boot application that manages data persistence using
Spring Data JPA with Hibernate ORM, connecting to a PostgreSQL database.

1. Core Dependencies (from backend/pom.xml):
   - ORM framework: spring-boot-starter-data-jpa (Hibernate under the hood)
   - JDBC Driver: org.postgresql:postgresql (v42.7.3)
   - Jackson Integration: jackson-datatype-hibernate6 (prevents lazy serialization errors)
   - Boilerplate Reduction: org.projectlombok:lombok

2. ORM Concept of the Project:
   - Database tables are represented as Java classes marked with the `@Entity` annotation.
   - Primary Keys are mapped via `@Id` and automatically generated with the IDENTITY strategy.
   - DB columns are mapped to class fields, with custom names provided by `@Column(name = "...")`.
   - Relationships between tables are declared using annotations such as `@ManyToOne`,
     `@OneToMany`, `@JoinColumn`, and configured with cascading options and lazy/eager loading.
   - Fields that are computed dynamically and should not be saved in the database tables
     are marked with the `@Transient` annotation.
   - Database queries are declared in Interface repositories extending `JpaRepository<Entity, IdType>`.
     These queries are either:
       a) Automatically generated by Spring Data JPA from the method name convention.
       b) Defined using custom JPQL (Java Persistence Query Language) via `@Query("...")`.

Below is a full breakdown of all Spring Boot REST API endpoints, grouped by

--------------------------------------------------------------------------------
1. AuthController (Base Route: `/api/auth`)
--------------------------------------------------------------------------------
* Class File: backend/src/main/java/com/example/terraspoter/controller/AuthController.java
* Endpoints:
  - `GET /api/auth/`
    * Description: Health check and asynchronous warm-up request to the ML recommendation microservice.
  - `POST /api/auth/send-otp`
    * Request Body: `{ email, fname }`
    * Description: Validates email uniqueness for signup, then emails a 4-digit verification OTP.
  - `POST /api/auth/signup`
    * Request Body: `SignupRequest` (`{ fname, lname, email, phoneNo, dob, password, otp }`)
    * Description: Validates OTP, creates the user account in the database (with BCrypt password hashing), and triggers a welcome email.
  - `POST /api/auth/login`
    * Request Body: `LoginRequest` (`{ email, password }`)
    * Description: Authenticates user credentials, sets session max inactive interval (30 min), and binds `userId` to the session.
  - `GET /api/auth/session`
    * Description: Validates active session and returns profile data of the logged-in user.
  - `POST /api/auth/logout`
    * Description: Invalidates the active HttpSession.
  - `POST /api/auth/google`
    * Request Body: `{ email, fname, lname }`
    * Description: Performs Google SSO authentication (automatic registration for new users).
