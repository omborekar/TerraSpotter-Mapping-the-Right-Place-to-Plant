TerraSpotter — Backend API & Code Overview

Location: backend/

Purpose
-------
This document describes the backend Java (Spring Boot) codebase: controllers (HTTP API endpoints), services (business logic), repositories (DB access), models (JPA entities), DTOs/payloads, and configuration. It lists each important file, the endpoints or methods it exposes, expected request payloads, response shapes, and where external integrations (ML, Cloudinary, Brevo) are used.

How to run
----------
From the `backend` folder:

  - Linux / macOS:
      ./mvnw spring-boot:run
  - Windows (PowerShell / cmd):
      mvnw.cmd spring-boot:run

The backend listens on `server.port` (default 8080). Environment variables used are documented below.

Environment / application properties
-----------------------------------
Key properties are defined in `backend/src/main/resources/application.properties` and use environment variables:

  - DB_URL, DB_USER, DB_PASS — PostgreSQL JDBC connection
  - PORT — server port (default 8080)
  - CLOUDINARY_CLOUD_NAME, CLOUDINARY_API_KEY, CLOUDINARY_API_SECRET — Cloudinary credentials
  - BREVO_API_KEY, BREVO_SENDER_EMAIL, BREVO_SENDER_NAME — Brevo (email) credentials
  - ML_API_URL — base URL for the ML recommendation service used by `LandService`

Other relevant configured properties:
  - `spring.jpa.hibernate.ddl-auto=update`
  - Session timeout: 30m
  - Multipart limits: max file size 20MB, max request 100MB

Controller endpoints (HTTP API)
------------------------------
Files: `backend/src/main/java/com/example/terraspoter/controller/*.java`

1) AuthController.java — `backend/src/main/java/com/example/terraspoter/controller/AuthController.java`
   - Base path: `/api/auth`
   - GET `/api/auth/`  — health-check + ML warmup (returns a small string). No auth.

   - POST `/api/auth/send-otp`
     - Body: JSON { "email": "user@example.com", "fname": "Optional" }
     - Behaviour: checks if email already registered via `AuthService`. If not, generates 4-digit OTP and calls `BrevoEmailService.sendOtpEmail`. Stores OTP in `OtpStore` (in-memory).
     - Responses: 200 OK { message }, 400 for missing email, 409 if email exists, 500 on email send failure.

   - POST `/api/auth/signup`
     - Body: `SignupRequest` (JSON):
       { "fname","lname","email","phoneNo","dob" (yyyy-MM-dd),"password","otp" }
     - Behaviour: validates OTP via `OtpStore.validate()`, checks duplicate email, hashes password via `AuthService.saveUser()` and persists `User`. Sends welcome email (non-fatal on error).
     - Responses: 200 OK on success, 422 for invalid/expired OTP, 400 on validation errors.

   - POST `/api/auth/login`
     - Body: `LoginRequest` (JSON): { "email", "password" }
     - Behaviour: verifies credentials via `AuthService.checkPassword()`. On success sets `HttpSession` attribute `userId` and returns `User` (with `password` nulled). Session timeout = 30 min.
     - Responses: 200 OK (User JSON), 401 Unauthorized on wrong credentials.

   - GET `/api/auth/session`
     - Behaviour: returns current session user via `userId` in `HttpSession` or 401 if none.

   - POST `/api/auth/logout`
     - Behaviour: invalidates the session.

   - POST `/api/auth/google`
     - Body: JSON { "email", "fname", "lname" }
     - Behaviour: simple Google login/register helper (creates user if absent) and sets session.

2) UserController.java — `backend/src/main/java/com/example/terraspoter/controller/UserController.java`
   - Base path: `/api/users`
   - GET `/api/users/profile` — reads `userId` from `HttpSession`, returns `User` entity (or 401 if not authenticated).
   - PUT `/api/users/profile` — Body: JSON map of fields to update (Map<String,String>), supported keys: `fname`, `lname`, `phoneNo`. Updates the `User` record for session user.
   - Note: CORS is configured on this controller to allow the deployed frontend origin.

3) StatsController.java — `backend/src/main/java/com/example/terraspoter/controller/StatsController.java`
   - Base path: `/api`
   - GET `/api/stats` — returns aggregated statistics from `StatsService` as a Map: { totalLands, approvedLands, treesPlanted, volunteers }

4) LandController.java — `backend/src/main/java/com/example/terraspoter/controller/LandController.java`
   - Base path: `/api/lands`
   - GET `/api/lands` — returns `List<Land>` (all lands)
   - GET `/api/lands/{id}` — returns detailed `Land` (transient fields attached: latest plantation start/completion details, totals)
   - POST `/api/lands` — Save a land record. Expects JSON payload (see "saveLand payload" below). Requires session `userId`. Returns saved `Land`.
   - POST `/api/lands/{id}/images` — Upload images. Multipart form, `files` param (List<MultipartFile>). Images are uploaded to Cloudinary via `CloudinaryService.uploadImage`. Stores `LandImage` records.
   - GET `/api/lands/{id}/images` — List `LandImage` objects for the land
   - GET `/api/lands/{id}/recommendations` — Returns recommendations (`List<LandRecommendation>`) from DB for the land
   - POST `/api/lands/{id}/recommendations/refresh` — Recomputes recommendations by calling ML API via `LandService.refreshRecommendations()`; returns fresh recommendations. Can return 502/500 on ML/network errors.
   - POST `/api/lands/{id}/plantation-start` — Begin plantation for a land. Body: JSON payload (see below). Requires session `userId`. Changes `landStatus` to "Under Plantation" and records a `PlantationStart` entity.
   - POST `/api/lands/{id}/plantation-complete` — Complete plantation. Expects request parameters (multipart/form-data): `treesPlanted` (int), optional `moreCapacity` (int), optional `notes` (string), optional `images` (files). Requires session `userId` to match the plantation owner. Persists `PlantationCompletion`, uploads images to Cloudinary (stored as `PlantationCompletionImage`), resets land to status "Vacant".
   - GET `/api/lands/{id}/reviews` — Returns `List<LandReview>`
   - POST `/api/lands/{id}/reviews` — Body: JSON map { rating, feasibilityNote, permissionNote, body } — saves/updates a `LandReview` for the session user.
   - GET `/api/lands/my` — Lands created by session user
   - GET `/api/lands/pending` — Returns lands with status = "PENDING" (via `LandRepository.findByStatus("PENDING")`)

5) LandVerificationController.java — `backend/src/main/java/com/example/terraspoter/controller/LandVerificationController.java`
   - Base path: `/lands` (note: not `/api/lands` here)
   - POST `/lands/{id}/verify` with query params `vote` (APPROVE/REJECT) and `userId` — calls `LandVerificationService.verifyLand(landId,userId,vote)` which updates `Land.status` from PENDING → APPROVED/REJECTED. Duplicate checks and counts are handled in repository/service.

6) PlantationController.java — `backend/src/main/java/com/example/terraspoter/controller/PlantationController.java`
   - Base path: `/api/plantations`
   - GET `/api/plantations/completed` — returns `List<PlantationShowcaseDTO>` (constructed from `PlantationCompletion` + images + reviews)
   - POST `/api/plantations/{id}/review` — Request params `rating` and `comment` (and `HttpSession` for user). Adds a review to a plantation completion.
   - GET `/api/plantations/completions/my` — intended to return completions by session user (calls `plantationService.getCompletionsByUser(userId)` in controller, but note: controller contains a likely bug/typo where `plantationService` variable is not declared — review required).

Services (business logic)
-------------------------
Files: `backend/src/main/java/com/example/terraspoter/service/*.java`

- `AuthService` — user management helpers:
  - `findUserById(Long)`, `findUserByEmail(String)`, `saveUser(User)` (hashes password with BCrypt), `checkPassword(raw, encoded)`.

- `OtpStore` — in-memory OTP store with TTL (10 minutes); methods `save(email, otp)`, `validate(email, otp)`, `invalidate(email)`.

- `BrevoEmailService` — sends transactional emails (OTP and welcome) using Brevo SMTP API. Configured with `brevo.api-key`, `brevo.sender-email`, `brevo.sender-name`.

- `CloudinaryService` — uploads images to Cloudinary and returns secure URLs. Uses `Cloudinary` bean from `CloudinaryConfig`.

- `LandService` — core business logic for lands. Key methods and behaviour:
  - `getAllLands()`, `getLandsByUser(userId)`, `getLandById(id)` — attaches transient plantation start/completion details using the `PlantationStartRepository` and `PlantationCompletionRepository`.
  - `saveLand(Map<String,Object> payload, Long userId)` — builds a `Land` entity from the payload (`buildLandFromPayload`) and persists it; triggers `fetchAndSaveRecommendations(saved)` which calls the external ML API and persists `LandRecommendation` rows.
  - `saveImages(Long landId, List<MultipartFile> files)` — uploads each file to Cloudinary and creates `LandImage` rows.
  - `startPlantation(Long landId, Map payload, Long userId)` — validates status, creates a `PlantationStart` record using payload fields `plannedDate` (String yyyy-MM-dd), `teamSize`, `treesToPlant`, `method`, `notes`.
  - `completePlantation(Long landId, Integer treesPlanted, Integer moreCapacity, String notes, List<MultipartFile> images, Long userId)` — validates user has permission, saves `PlantationCompletion`, uploads completion images to Cloudinary, saves `PlantationCompletionImage` rows, resets land status.
  - `getReviews(Long landId)` and `addReview(Long landId, Map payload, Long userId)` — handles `LandReview` creation/upsert.
  - `refreshRecommendations(Long landId)` — orchestrates Open-Meteo API calls to derive `temp`, `rainfall`, `soil`, `climate` inputs; calls the ML API at `${ML_API_URL}/predict?temp=...&rainfall=...&soil=...&climate=...`; parses `recommendations` array from ML response into `LandRecommendation` entities and persists them. On ML failure a fallback set of recommendations is saved.

  Implementation notes: `buildLandFromPayload` expects the land payload structure documented below.

- `PlantationShowcaseService` — constructs `PlantationShowcaseDTO` objects by joining `PlantationCompletion`, its images, related `Land` info and `User` (team name). Also supports adding reviews on completions.

- `StatsService` — aggregates counts for frontend dashboard: total lands, approved lands, total trees planted (custom query on `PlantationCompletionRepository`), volunteers count (users).

Repositories (DB access)
------------------------
Files: `backend/src/main/java/com/example/terraspoter/repository/*.java`

Key repository interfaces and notable methods:

- `UserRepository` — extends `JpaRepository<User,Long>`
  - `Optional<User> findByEmail(String email)`
  - `@Query("SELECT COUNT(DISTINCT(u.email)) FROM User u") long userCount()`

- `LandRepository` — `JpaRepository<Land,Long>`
  - `List<Land> findByStatus(String status)`
  - `List<Land> findByCreatedBy(Long userId)`
  - `long countByStatus(String status)`
  - `@Query("SELECT SUM(l.areaSqm) FROM Land l WHERE l.status = :status") Double sumAreaSqmByStatus(String status)`

- `LandImageRepository` — `List<LandImage> findByLandId(Long landId)`
- `LandRecommendationRepository` — `List<LandRecommendation> findByLandId(Long)` and `void deleteByLandId(Long)`
- `LandReviewRepository` — `List<LandReview> findByLandIdOrderByCreatedAtDesc(Long)`, `Optional<LandReview> findByLandIdAndUserId(Long, Long)`
- `LandVerificationRepository` — `boolean existsByLandIdAndUserId(Long,Long)`, `long countByLandIdAndVote(Long,String)`
- `PlantationStartRepository` — `Optional<PlantationStart> findTopByLandIdOrderByCreatedAtDesc(Long)`
- `PlantationCompletionRepository` — `Optional<PlantationCompletion> findTopByLandIdOrderByCreatedAtDesc(Long)`, `List<PlantationCompletion> findByLandIdOrderByCreatedAtDesc(Long)`, and `@Query("SELECT COALESCE(SUM(p.treesPlanted), 0) FROM PlantationCompletion p") Long totalTreesPlanted();`
- `PlantationCompletionImageRepository` — standard JpaRepository for completion images

Models (entities) — summary
---------------------------
Files: `backend/src/main/java/com/example/terraspoter/model/*.java`
Important JPA entities and main fields (persisted columns shown):

1) `User` (`users` table)
   - id (Long)
   - fname, lname, email, phoneNo, dob (LocalDate)
   - role (String)
   - password (stored, marked `@JsonIgnore` so not serialized)

2) `Land` (`land_details`)
   - id, verified (Boolean), district, title, description
   - polygonCoords (JSONB column) — stored as JSON string
   - centroidLat, centroidLng (Double), areaSqm
   - ownerName, ownerPhone, ownershipType, permissionStatus
   - landStatus / status (String) — e.g., PENDING, APPROVED, Under Plantation, Plantation Complete, Vacant
   - waterAvailable, waterFrequency, fencing (Boolean), notes, accessRoad, soilType, nearbyLandmark
   - createdBy (userId), createdAt (timestamp)
   - Transient fields (not persisted): plantationDetail (PlantationStartDetail), completionDetail (PlantationDoneDetail), remainingCapacity, totalTreesPlanted, totalRounds

3) `LandImage` (`land_images`)
   - id, landId, imageUrl, createdAt

4) `LandRecommendation` (`land_recommendations`)
   - id, landId, plantName, suitabilityScore (Double), reason (string), createdAt

5) `LandReview` (`land_reviews`)
   - id, landId, userId, rating (Integer), feasibilityNote, permissionNote, body (text), createdAt
   - Transient userName populated on read

6) `LandVerification` (`land_verifications`)
   - id, landId, userId, vote ("APPROVE"/"REJECT"), createdAt
   - Unique constraint on (land_id, user_id)

7) `PlantationStart` (`plantation_starts`)
   - id, landId, userId, plannedDate (LocalDate), teamSize, treesToPlant, method, notes, createdAt

8) `PlantationCompletion` (`plantation_completions`)
   - id, landId, userId, treesPlanted, moreCapacity, notes (TEXT), createdAt
   - relations: `List<PlantationCompletionImage> images` (OneToMany mappedBy completionId)
   - relations: `List<PlantationReview> reviews` (OneToMany)
   - Transient helper fields: landTitle, location, teamName

9) `PlantationCompletionImage` (`plantation_completion_images`)
   - id, completionId (ManyToOne -> PlantationCompletion), imageUrl (TEXT), createdAt

10) `PlantationReview` and `PlantationReviewImage` — reviews on completions and attached images.

ML integration
--------------
- `LandService` calls external weather APIs (Open-Meteo) to derive mean temperature, annual rainfall, and soil moisture.
- It then builds ML input and calls `${ML_API_URL}/predict?temp=...&rainfall=...&soil=...&climate=...` (HTTP GET) — expects JSON response with `recommendations` array.
- Expected ML response shape (example):
{
  "recommendations": [
    { "tree": "Neem", "confidence": 0.72, "reasons": ["drought resistant", "native"] },
    ...
  ]
}
- If ML fails, `LandService` saves fallback recommendations.

Cloudinary & images
-------------------
- `CloudinaryConfig` creates a `Cloudinary` bean using `cloudinary.cloud-name`, `cloudinary.api-key`, `cloudinary.api-secret`.
- `CloudinaryService.uploadImage(MultipartFile,fileFolder)` uploads and returns `secure_url`. The backend stores image URLs in `land_images` and `plantation_completion_images` tables.

Email (Brevo)
--------------
- `BrevoEmailService` posts to Brevo API to send OTP and welcome emails. Requires `brevo.api-key`, `brevo.sender-email`, `brevo.sender-name`.

Security & CORS
---------------
- `SecurityConfig` provides a BCrypt `passwordEncoder()` bean and CORS configuration.
- Security rules permit all requests to `/api/auth/**` and `/api/lands/**` by default (the config currently permits many endpoints, review for production security).
- Sessions: server uses cookie-based sessions; `User` is tracked by storing `userId` in `HttpSession`.

Notes / TODOs (quick developer hints)
------------------------------------
  - `PlantationController` contains an import and an undeclared variable (`plantationService`) — may cause compilation errors; review and fix.
  - Many endpoints are currently `permitAll()` in `SecurityConfig` — tighten for production and apply role-based access where needed.
  - Image upload handling stores Cloudinary `secure_url`. If you need to remove or replace images, `CloudinaryService.deleteImage` is available but requires `publicId` (not currently stored separately).
  - `OtpStore` is in-memory: OTPs will be lost on backend restart. For production use, persist OTPs or use a dedicated store with expiry.
